## Bugs Found

### Bug 1 — QA rerun used stale Seller-era selectors

- Severity: medium for QA readiness, not product runtime.
- Where: `scripts/run-clean-rerun.mjs`.
- Current behavior before fix: `qa:clean-rerun` failed waiting for `Value tied to one booked appointment`.
- Expected behavior: clean rerun should follow the current Sprint 01 onboarding copy.
- Cause: QA script still referenced old booking/Seller-era labels.
- Fix applied:
  - Updated `Value tied to one booked appointment` to `Estimated appointment value`.
  - Updated final activation button matcher from `go live with .*seller` to `activate revory read`.
- Status: fixed.

### Bug 2 — Landing FAQ email link pointed to Cloudflare obfuscation route

- Severity: small but dangerous for trust.
- Where: `src/content/revory-landing-reference.html`; normalized defensively in `src/app/page.tsx`.
- Current behavior before fix: rendered `Email us` link pointed to `/cdn-cgi/l/email-protection...`, producing a `404`.
- Expected behavior: public contact affordance should not point to a missing internal route.
- Cause: legacy exported/reference landing HTML contained Cloudflare-style email protection URLs.
- Fix applied:
  - Replaced rendered email-protection links with `mailto:hello@revory.com`.
  - Updated the HTML source reference to remove the broken `/cdn-cgi` links.
  - Kept a defensive normalization in the landing adapter.
- Status: fixed.

No active public promise was found claiming:

- confirmed lost revenue;
- guaranteed recovery;
- revenue generated by REVORY;
- full CRM/inbox/BI capability;
- AI sales assistant behavior.

## Overpromise Risks

- The current product still uses transitional booking-opportunity substrate. That is acceptable only while copy keeps it framed as **blocked booking opportunity risk** or **bounded action guidance**.
- Full leak engine language should remain controlled until `RevenueLeak` modeling / deterministic leak calculations are implemented.
- The product must keep using **estimated revenue at risk**, not confirmed loss.

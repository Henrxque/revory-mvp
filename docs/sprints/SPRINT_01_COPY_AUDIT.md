# Sprint 01 Copy Audit — Product Repositioning & Source of Truth Alignment

## Summary

The current codebase still presents REVORY primarily as **REVORY Seller**, a booking acceleration product for turning paid leads into booked appointments. That framing is still visible across public metadata, landing/reference copy, pricing, auth, app shell, onboarding, setup, dashboard, Daily Brief, Booking Inputs, Booking Assistance, billing copy, password reset email, legal pages, decision-support copy, and LLM prompt instructions.

For Sprint 01, the highest-risk issue is not technical implementation. It is positioning drift: the UI still teaches the buyer/user that the product is a **Seller / booking-first / booked proof / booking assistance** system instead of a **Revenue Leak Detector for premium MedSpas**.

The public brand should become **REVORY**. The category should become **Revenue Leak Detector for premium MedSpas**. Financial language must use **estimated revenue at risk**, not confirmed lost revenue, not revenue generated by REVORY, and not causal attribution.

No `RevenueLeak` model or leak engine should be implemented in this step. This audit only identifies copy and framing that need to change.

### Core promise

- To: `Detect where your MedSpa is leaking revenue and what to fix first`

- To: `Estimated Revenue at Risk This Month`

- To: `Daily Leak Brief`

- To: `Executive Revenue Leak Summary`

## Risks

- If public copy is not changed first, buyers will still evaluate REVORY as booking acceleration software, not a Revenue Leak Detector.
- If `booked proof` remains the center of the app, V3 will feel like a cosmetic rebrand instead of a real product repositioning.
- If `Seller` remains in onboarding/setup, users will learn the old mental model during activation.
- If decision-support and LLM prompts stay Seller/booking-first, generated guidance may fight the new UI copy.
- If revenue copy is changed too aggressively without leak engine support, the product could imply confirmed lost revenue before deterministic evidence exists.
- If booking assistance is removed instead of reframed, the V3 loses useful substrate for `missing contact leak risk` and `booking path blocked risk`.
